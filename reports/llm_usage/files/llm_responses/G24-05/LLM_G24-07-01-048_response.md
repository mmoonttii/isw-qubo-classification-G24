Final delivery:**What changed, and what I verified:**

**1. Dtype-typed reads (both `train()` and `predict()`).** Feature columns now read as float32 instead of pandas' float64 default. This isn't just a memory nicety here — sklearn's tree code (`RandomForestClassifier`, `GradientBoostingClassifier`) internally casts `X` to float32 regardless (its `DTYPE` constant), so handing it float64 previously forced an extra internal copy/cast that float32 input skips.

One thing I deliberately did *not* do: force the target column to an int dtype at parse time. The spec allows `predict()` to run against "any other reduced file," not just this pipeline's own output — and such a file could plausibly store `0`/`1` as `"1.0"`/`"0.0"` text, which an `int8` parse dtype would reject outright. I tested this exact scenario and it now works correctly; forcing int8 at parse time would have broken it.

**2. `train()`: `df.pop()` instead of `df.drop(columns=...)`.** The old `drop()` allocates a brand-new DataFrame with every column except the target, so the original frame and the new one are briefly both alive — effectively a second copy of the feature matrix. `pop()` removes the target in place. Measured on a synthetic 1.05M-row × 29-feature set (realistic post-selection size for a 1.5M-row source): **peak RSS 547 MB → 312 MB**.

**3. `predict()` rewritten to stream.** This was the real target, since test-set inference is what the docente's 1.5M+ row verification set exercises directly. It now reads in 200k-row chunks, predicts per chunk, and appends to `predictions.csv` incrementally — never holding the full test feature matrix, the full prediction arrays, *and* the full output DataFrame simultaneously the way the old version did.
